# Subscription Event Key-Based Registration System Implementation Plan

## 1. Objective
Replace the `allowedParticipants` UID whitelist mechanism with unique 10-digit one-time-use numeric keys for subscription event registration, enabling secure yet decentralized access control.

## 2. Impact Analysis

### Files to Modify
- **firestore.rules** - Add security rules for keys subcollection
- **app.js** - Main implementation (remove old functions, add new key management system)

### Files NOT to Create
No new files needed; all changes are within existing files.

### Dependencies
- Firebase Firestore: `runTransaction()` for atomic operations
- Existing: `SyncManager` for caching, auth, serverTimestamp
- No external libraries required

### Security Considerations
- **Prevention of key enumeration**: Use document ID lookup (read specific doc) instead of collection listing
- **Atomic operations**: Use Firestore transactions to prevent race conditions (multiple students claiming same key)
- **One-time use guarantee**: Field `used: boolean` checked within transaction
- **Firestore Rules**: Custom rules deny listing `/subscription_events/{eventId}/keys` collection

### Risk Assessment
| Risk | Mitigation |
|------|-----------|
| Race condition on key claim | Use Firestore `runTransaction()` for atomic check-and-claim |
| Key guessing attacks | 10-digit numeric keys = 10 billion possibilities; rate-limit registration attempts if needed |
| Key theft via Firestore export | Rules prevent collection queries; keys only readable individually |
| Data migration burden | `allowedParticipants` remains in existing events; gradual deprecation possible |

## 3. Implementation Details

### Phase 1: Update Firestore Rules (firestore.rules)
**Objective**: Add security rules for the new `/subscription_events/{eventId}/keys/{keyId}` subcollection.

**Changes**:
- Add a `match /subscription_events/{eventId}/keys/{keyId}` block
- Allow authenticated users to **read a specific key document** (by ID): `allow read: if isSignedIn() && request.path.size() == 6` (prevents collection-level queries)
- Allow admins to **write/create keys**: `allow write: if isAdmin()`
- Key fields must validate: `keyId` must be numeric and 10 digits

**Validation logic in rules**:
```
- keyId matches pattern: /^\d{10}$/
- Document structure: { used: boolean, usedBy: string|null, usedAt: timestamp|null, createdAt: timestamp }
```

### Phase 4: Implement Key Claiming Logic (app.js - New Function)
**Function name**: `mcClaimSubEventKey(eventId, keyValue, uid)`

**Logic** (using Firestore transaction):
```
BEGIN TRANSACTION
  1. Read /subscription_events/{eventId}/keys/{keyValue}
  2. Check if document exists
  3. Check if field `used === false`
  4. If both pass:
     - Update `used = true, usedBy = uid, usedAt = serverTimestamp()`
     - Return { success: true, message: "Key claimed" }
  5. If document doesn't exist or already used:
     - Abort transaction
     - Return { success: false, message: "Invalid or already-used key" }
COMMIT or ABORT
```

**Error handling**:
- Invalid key (doesn't exist) → "Invalid key. Please check and try again."
- Already used → "This key has already been used."
- Network error → "Failed to verify key. Check your connection."

### Phase 5: Implement Admin Key Generation (app.js - New Function)
**Function name**: `mcGenerateSubEventKeys(eventId, quantity)`

**Logic**:
1. Generate `quantity` unique 10-digit numeric keys
2. Batch write to Firestore: create documents at `/subscription_events/{eventId}/keys/{keyId}` with:
   ```javascript
   {
     used: false,
     usedBy: null,
     usedAt: null,
     createdAt: serverTimestamp()
   }
   ```
3. Use Firestore `writeBatch()` for efficiency (max 500 per batch)
4. If quantity > 500: loop with multiple batches
5. Return array of generated keys for admin to export

**Collision handling**:
- Generate candidate key
- Check if document exists
- If exists (unlikely with 10 billion possibilities), regenerate
- Retry max 3 times per key; if collision persists, skip that seed and try next

## 5. Verification Strategy

### Unit Testing Approach
1. **Key Generation**:
   - Generate 1000 keys; verify all unique
   - Verify all keys are exactly 10 numeric digits
   - Verify Firestore documents created with correct schema

2. **Key Claiming**:
   - Valid unused key → Success, document marked `used=true`
   - Valid but already-used key → Failure, no state change
   - Invalid key (non-existent) → Failure
   - Concurrent claims (2+ simultaneous) → Only 1 succeeds; others fail cleanly

3. **Registration Flow**:
   - Wrong key → Rejects, can retry
   - Correct key → Advances to subject selection
   - Subject selection → Registration saved with key reference

### Integration Testing
- Admin generates keys for event A
- Create 3 test user accounts
- Each user enters different keys (1 valid, 1 invalid, 1 already-used)
- Verify only valid key allows progression
- Verify registration created with correct key reference
- Verify Firestore document shows `used=true, usedBy=uid, usedAt=timestamp`

### Acceptance Criteria
- ✅ Key generation: batch creates 100+ unique 10-digit keys in < 2s
- ✅ Key claiming: transaction succeeds for first claimant, fails for subsequent attempts
- ✅ Admin UI: displays key stats, generation form, export options
- ✅ Student UI: two-step modal with key entry then subject selection
- ✅ Error messages: clear and actionable for both success and failure cases
- ✅ Firestore rules: prevent key collection enumeration; allow individual key reads
- ✅ No `allowedParticipants` logic remains in codebase
- ✅ Existing events can still be managed (graceful handling of old events without keys)

## Summary
This implementation replaces a centralized whitelist (allowedParticipants) with a decentralized one-time-use key system. The key design leverages Firestore's granular security model to prevent key enumeration while enabling efficient individual key verification. The admin interface provides intuitive key generation and management tools, while the student experience is streamlined with a simple two-step registration modal.
